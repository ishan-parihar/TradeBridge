# ADR-003: Canonical Ownership and Idempotency Contract

## Status

Proposed

## Date

2026-04-06

## Context

The TradeBridge system operates in environments where multiple agents, strategies, or human traders may share a single MT5 account. Without a consistent ownership vocabulary and idempotency contract, the system cannot:

- Distinguish its own orders, positions, and P&L from foreign activity
- Safely retry failed writes without creating duplicates
- Provide agents with an authoritative view of "what is mine"
- Support demo → live → shared-account migration without re-architecting identity

This ADR records the canonical meaning of every identity field used across schemas, tools, prompts, EA execution, and journaling. It is the authoritative reference for Phases 1.x (ownership wiring, idempotency enforcement, reconciliation) and all subsequent roadmap phases.

## Decision

### 1. Canonical identity field definitions

| Field | Owner | Scope | Type | Generated by | Immutable after write |
|---|---|---|---|---|---|
| `session_id` | Agent runtime | Single agent run / invocation window | `str` (UUID or stable opaque string) | Agent at session start | No — each new session gets a new value |
| `strategy_id` | Strategy configuration | All sessions using the same strategy | `str` (stable namespace identifier) | Operator / deploy config | Yes — identifies the strategy itself |
| `intent_id` | Agent (per trade) | One trade intent from discovery to exit | `str` (unique per intended trade) | Agent before submission | Yes — represents a single decision to trade |
| `decision_id` | Journal service | One journal record lifecycle (entry + exit) | `str` (UUID, auto-assigned) | `trading/log_decision` on first log | Yes — journal records are append-only |
| `magic_number` | Platform / EA | MT5-level ownership filtering for orders/positions | `int` (numeric namespace) | Derived from `strategy_id` or assigned per strategy config | Yes — embedded in MT5 order ticket |
| `comment` | Platform / Agent | Human-readable breadcrumb in MT5 order/position comment field | `str` (truncated to MT5 limit) | Platform, composed from `intent_id`, `session_id`, `strategy_id` | Yes — written once with the order |
| `idempotency_key` | Agent / Gateway | Write-path deduplication across retries and failover | `str` (opaque, deterministic per intent) | Agent or gateway, stable across retries | Yes — same key = same write |

### 2. Field semantics and contracts

#### `session_id` — Session/run ownership scope

- Identifies a single agent invocation window.
- All writes produced during one run share the same `session_id`.
- Used to:
  - Separate trades from this run vs prior runs in the journal
  - Attribute positions and orders when reconciling after restart
  - Filter `trading/reflect` queries to "this session only"
- **Not a strategy identifier.** A strategy may produce many sessions over time.
- **Not stable across restarts.** A new agent run gets a new `session_id`.

#### `strategy_id` — Stable strategy namespace

- Identifies the strategy logic / configuration in use.
- Shared across all `session_id` values that run the same strategy.
- Used to:
  - Group trades by strategy for performance attribution
  - Configure per-strategy policy limits (max risk, max trades, symbol whitelist)
  - Derive `magic_number` for MT5-level ownership filtering
- **Must be stable** across deploys and restarts for the same strategy.
- Examples: `"xau-breakout-h1"`, `"btc-bracket-m15"`, `"eur-pullback-daily"`.

#### `intent_id` — One trade intent

- Uniquely identifies a single intended trade from conception through fill/exit/cancel.
- Used to:
  - Prevent duplicate submissions (the same `intent_id` must not produce two orders)
  - Link execution results back to the originating decision
  - Enable idempotent retries: same `intent_id` + same parameters = no-op if already filled
- **Must be generated before any submission attempt.**
- Must be unique within the account namespace for the lifetime of the order/position.
- Recommended format: `{strategy_id}-{symbol}-{side}-{timestamp}` or UUID.

#### `decision_id` — Journal record lifecycle key

- Auto-assigned by the journal service when `trading/log_decision` is first called for a trade.
- Links the entry-side journal record to the exit-side update.
- Used to:
  - Join execution history with decision reasoning
  - Query "what was I thinking when I entered this trade?"
  - Compute decision quality metrics separate from outcome
- **Not the same as `intent_id`.** `intent_id` is the agent's trade identifier; `decision_id` is the journal's.
- A single `intent_id` maps to exactly one `decision_id`.

#### `magic_number` — Numeric strategy namespace for MT5 ownership filtering

- An integer embedded in every MT5 order and position via the EA.
- Used to:
  - Filter orders/positions by strategy at the MT5 terminal level
  - Separate owned vs foreign activity on shared accounts
  - Enable EA-native trailing stop and bracket managers to only touch owned positions
- **Derived from `strategy_id`**, either via:
  - A configured mapping (`strategy_id → magic_number` in config)
  - A deterministic hash (if no explicit mapping exists)
- Each strategy must have a unique `magic_number` within a single account.
- MT5 constraint: `magic_number` is a 32-bit unsigned integer (`0` to `4294967295`).
- `0` is reserved for untagged / foreign activity.

#### `comment` — Human-readable breadcrumb

- A string placed in the MT5 order/position comment field.
- Used to:
  - Provide at-a-glance attribution in the MT5 terminal UI
  - Assist manual reconciliation when terminal and journal disagree
  - Serve as a fallback ownership signal when `magic_number` alone is insufficient
- **Composed by the platform** from identity fields, typically:
  ```
  {strategy_id}:{intent_id}:{session_id}
  ```
  Truncated to MT5's comment length limit (implementation-defined, typically 31 characters).
- Must not contain sensitive information or free-form text from the agent rationale.
- Deterministic: same intent produces the same comment.

#### `idempotency_key` — Write-path deduplication contract

- An opaque string that uniquely identifies a single write operation across retries, timeouts, and transport failover.
- Used to:
  - Prevent duplicate order submission when the agent retries after a timeout
  - Prevent double-enqueue during TCP → HTTP failover
  - Enable safe replay of failed writes
- **Must be stable across all retry attempts** for the same logical write.
- Recommended derivation: deterministic hash of (`intent_id`, `symbol`, `side`, `volume_lots`, `order_kind`, `price` if pending).
- Enforced at:
  1. Gateway queue layer (`gateway_queue.py`) — dedupe on enqueue
  2. Execution gateway — reject duplicate keys within TTL window
  3. Adapter layer — pass through to EA or direct adapter
- TTL: configurable, default 24 hours (covers agent restart and reconnection scenarios).

### 3. Ownership propagation model

Every write operation (market order, pending order, modify, close, cancel) must carry the full identity tuple:

```
(session_id, strategy_id, intent_id, magic_number, comment, idempotency_key)
```

Propagation chain:

```
Agent → MCP Server → Execution Gateway → Queue → Adapter → EA → MT5
```

At each layer:
- **MCP Server**: validates presence of identity fields on write tool requests
- **Execution Gateway**: assigns `idempotency_key` if not provided; checks dedup
- **Queue**: stores `idempotency_key` for dedup; passes identity through
- **Adapter**: maps `strategy_id → magic_number`; composes `comment`
- **EA**: embeds `magic_number` and `comment` in MT5 order; returns ownership metadata where possible
- **MT5**: stores `magic_number` and `comment` on order/position; exposes them in terminal queries

### 4. Idempotency enforcement contract

| Scenario | Behavior |
|---|---|
| Same `idempotency_key` submitted twice within TTL | Second submission returns cached result; no duplicate order |
| TCP failover during submission, HTTP retry with same key | Queue dedup prevents double-enqueue |
| Agent restart, retry of unconfirmed intent with same key | Gateway returns prior result or re-queues without duplication |
| Different `intent_id` but identical trade parameters | Treated as separate intent; both may execute |
| Same `intent_id` with different parameters | Rejected — parameters must match original or be a new intent |

### 5. Demo / live / shared-account migration assumptions

The following assumptions hold across all deployment environments:

| Assumption | Demo | Live | Shared Account |
|---|---|---|---|
| Ownership filtering required | Yes — prepare for live | Yes — critical | Yes — multiple agents/humans |
| `magic_number` enforced | Yes — same contract | Yes — same contract | Yes — must partition namespaces |
| `idempotency_key` required | Yes — build discipline | Yes — production safety | Yes — prevents cross-agent collisions |
| Foreign activity expected | No (or synthetic) | No (or operator manual) | Yes — treat all as foreign until attributed |
| Reconciliation before trade | Recommended | Mandatory | Mandatory |
| Journal attribution | Session-scoped | Session-scoped | Must filter by `strategy_id` + `session_id` |

#### Shared-account rule (canonical)

> On any account where foreign activity may exist, **all positions, orders, deals, and P&L are treated as foreign until attributed to a known `intent_id`, `strategy_id`, or `magic_number`**. Agents must never learn from unlabeled P&L as if it were their own.

#### Migration path

1. **Demo-first**: All Phases 1-3 run exclusively on demo. Identity fields are enforced but false positives are low-risk.
2. **Live (dedicated)**: Same contract, single owner. Reconciliation simplifies since all activity is owned.
3. **Shared account**: `magic_number` becomes the primary MT5-level filter. `comment` provides secondary verification. Reconciliation must explicitly separate owned vs foreign before any trade decision.

### 6. Relationship to existing ADRs

- **ADR-001** (MCP is the only AI-facing interface): This ADR defines the identity contract that flows through the MCP interface.
- **ADR-002** (Latency optimization): Idempotency keys enable safe retries without performance penalty from duplicate detection at the broker level.
- **ADR-003** (this document): Ownership and idempotency are foundational — no subsequent phase should proceed without these contracts ratified.

## Consequences

### Easier

- Agents can definitively answer "is this position mine?" without heuristic matching
- Retries are safe by contract, not by luck or timing
- Shared-account trading becomes possible with clear ownership boundaries
- Reconciliation service (Phase 1.3) has deterministic identity fields to join on
- Journal attribution (Phase 4.1) can link execution to reasoning via `intent_id` → `decision_id`
- EA-native automation (Phase 3) can filter by `magic_number` without parsing comments

### Harder

- Every write tool must accept and propagate identity fields — no exceptions
- `magic_number` configuration must be maintained per strategy per account
- Comment truncation requires careful composition to stay within MT5 limits
- Idempotency state (key → result mapping) must be persisted across restarts
- Shared-account reconciliation adds computational overhead (must scan all activity and filter)

### Trade-offs accepted

1. **Slightly larger request payloads** — identity fields add ~100-200 bytes per write (acceptable)
2. **Configuration burden** — `strategy_id → magic_number` mapping must be maintained (mitigated by deterministic hash fallback)
3. **Idempotency TTL memory** — 24-hour dedup window requires persistent storage (mitigated by configurable TTL and pruning)

### Risks if not adopted

- Duplicate orders during retries or failover (already observed in live trading)
- Inability to distinguish agent's trades from foreign activity on shared accounts
- Journal contamination from foreign P&L (already observed — "foreign P&L confusion" mistake category)
- EA-native automation (trailing stops, brackets) touching foreign positions
- No deterministic way to reconcile state after agent restart

## Glossary

This section provides concise, linkable definitions for use in the roadmap and other documentation.

| Term | Definition |
|---|---|
| **Ownership** | The property that an order, position, deal, or journal entry was created by a specific agent session running a specific strategy. Established via `session_id`, `strategy_id`, `magic_number`, and `comment`. |
| **Attribution** | The process of determining whether an observed MT5 entity (order, position, deal) belongs to this agent's session or is foreign. |
| **Foreign activity** | Any order, position, deal, or P&L on the account that cannot be attributed to the current agent's `session_id` or `strategy_id`. |
| **Session** | A single agent invocation window, bounded by start and termination. Identified by `session_id`. |
| **Strategy** | A reusable trading logic/configuration, identified by `strategy_id`, that may produce many sessions over time. |
| **Intent** | A single trade decision from conception through resolution. Identified by `intent_id`. One intent → at most one order. |
| **Decision** | A journal record linking reasoning to execution. Identified by `decision_id`. One decision ↔ one intent. |
| **Magic number** | An MT5-level integer tag embedded in orders/positions for ownership filtering. Derived from `strategy_id`. |
| **Comment** | A human-readable MT5 order/position comment composed from identity fields (`{strategy_id}:{intent_id}:{session_id}`). |
| **Idempotency key** | An opaque string ensuring a write operation executes at most once, even across retries, timeouts, and transport failover. |
| **Idempotency** | The property that repeating the same write request (same key, same parameters) produces exactly one effect. |
| **Reconciliation** | The process of comparing observed MT5 state (account, orders, positions, deals) against the agent's intent ledger to determine truth. |
| **Degraded mode** | Operational state where some data sources disagree; trade cautiously with reduced frequency. |
| **Blind mode** | Operational state where current state cannot be trusted; stop entering new trades. |
| **Dedup** | Short for deduplication — preventing duplicate writes via `idempotency_key`. |

## Related Open Questions (from Roadmap §8)

1. **Should ownership live primarily in `magic_number`, `comment`, or both?** → **Both.** `magic_number` is the primary MT5-level filter (numeric, efficient). `comment` is the secondary human-readable breadcrumb (fallback, terminal UI visibility). See §3 Propagation Model.
2. **Should reconciliation persistence live in the journal DB or a new execution ledger?** → Deferred to a separate ADR before Phase 1.3 implementation. This ADR defines the identity fields that any ledger would join on.
3. **Should trailing/bracket automation be fully EA-native, or hybrid with Python fallback?** → Deferred to Phase 3 planning. This ADR ensures EA-native automation has `magic_number` for ownership filtering regardless of implementation choice.
4. **What sandboxing model is acceptable for Python research tools?** → Out of scope for this ADR.
5. **How much of the portfolio layer should be strategy-scoped vs account-global?** → Deferred to Phase 6. This ADR ensures `strategy_id` and `magic_number` provide the scoping mechanism.
